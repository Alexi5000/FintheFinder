# Demo Contract

The demo must prove the product path, not only the UI shell.

## Unconfigured Local Demo

```bash
npm install
npm run verify
npm run dev
```

Expected: `/` renders the workspace, provider status shows missing keys, `/settings` shows model defaults, and `/api/health` returns non-secret configuration status.

For credential-free product-path proof, run:

```bash
npm run evals:replay
```

Expected: the replay executes the real worker runtime and pipeline functions through deterministic in-memory adapters, proves the research -> approval -> reporting -> fenced publication path, and reports zero live OpenAI, Exa, or Supabase calls. This is orchestration proof, not live provider-quality proof.

## Configured Live Demo

1. Configure `.env` from `.env.example`.
2. Run Supabase migrations.
3. Start the web service and worker.
4. Use an existing Supabase Auth session or API bearer token, create a session, enqueue a run, review sources/claims/events, approve, generate a report, export markdown. Fin consumes Supabase Auth; it does not include a separate in-app auth product.
5. Run `npm run evals:persist` to record the offline fixture eval history for the configured environment.
6. Capture the final report-reader screenshot or video, note the completed reporting run ID, and keep the media file inside `docs/demo/artifacts/`.
7. Export the proof bundle from Supabase by final reporting run ID:

```bash
npm run demo:export -- --reporting-run-id <reporting-run-id> --media docs/demo/artifacts/session-detail.png --media docs/demo/artifacts/report-reader.png --update-benchmark
```

8. Confirm the generated `docs/demo/live-demo.json`, report export, run export, three-scenario live eval output, screenshots or video, per-stage costs, aggregate cost, token usage, search count, and `docs/BENCHMARK.md` Live Run Log row all refer to the same session/research/reporting/approval IDs. Then run:

```bash
npm run demo:record
npm run evals:live
```

`npm run demo:record` fails closed until every evidence field is finalized, internally consistent, and inspectable. The verifier rejects the example manifest, placeholder IDs, all-zero trace IDs, future dates, empty artifacts, missing benchmark references, missing expected-vs-actual live scenarios, failed live eval output, incomplete provenance hashes, and nonpositive cost evidence.

The manifest must point at local artifacts inside the repo for `reportExport`, `evalOutput`, `runExport`, screenshots/video files, and `benchmarkDoc`. The report must be nontrivial markdown with linked sources. The eval output is generated by `npm run demo:export` from the configured reporting run and must be JSON with `passed: true`, `mode: "live"`, `status: "ok"`, the same reporting run ID and trace ID, and at least three live scenarios with expected requirements and observed scores/issues/regressions. The run export must be session-level JSON containing the session ID, research run, reporting run, approval record, final review audit, aggregate cost, per-stage costs, model calls, token usage, and Exa search count. `docs/BENCHMARK.md` must include one real Live Run Log row that references the same manifest, prompt, session ID, research run ID, reporting run ID, approval ID, report export, eval output, run export, screenshots/video, cost, model, searches, and tokens.

`npm run demo:export` requires at least one `--media` artifact, refuses to export secret-like content, and records SHA-256 hashes for every report/eval/run/media artifact in manifest provenance. `npm run demo:record` rejects missing, stale, or extra hashes, and `npm run evals:live` re-fetches Supabase evidence for the exported reporting run before accepting the proof.

On success, `npm run evals:live` re-fetches Supabase evidence for the exported reporting run, verifies the reporting run metadata links to the approval and source research run, checks the final review audit and `report_ready` event are bound to that reporting run, compares Supabase evidence with the manifest and local artifacts, then prints the session ID, research run ID, reporting run ID, approval ID, reporting trace ID, manifest SHA-256, eval scenario count, issues/regressions arrays, and the exact manifest cost object so the terminal proof can be compared with the benchmark row.

Do not claim live demo proof from the example manifest.
`npm run evals:live` also fails closed unless provider credentials, exporter provenance, Supabase evidence, and the recorded live scenario artifact exist.
